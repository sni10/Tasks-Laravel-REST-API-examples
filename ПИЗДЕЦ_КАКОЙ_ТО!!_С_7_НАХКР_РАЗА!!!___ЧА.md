### Коротко: как с первого раза получить нужный перевод без «танцев»

1) Скажите явно ЧТО переводить и ГДЕ: перечислите конкретные файлы/разделы (например: `README.md → только заголовки` или `README.md → весь текст кроме код-блоков`).
2) Укажите ЯЗЫКОВУЮ политику: «заголовки — EN, тело — RU» или «всё — RU» и т. п.
3) Дайте ГЛОССАРИЙ терминов (что переводить/не переводить: Production, Staging, CI/CD, tests и т. п.).
4) Определите СТИЛЬ: официальный/нейтральный, без жаргона, выравнивание терминов.
5) Уточните, что НЕ трогаем: кодовые блоки, HTTP-ответы, команды, форматирование Markdown, маршруты и пути.
6) Попросите ПРЕДПРОСМОТР: «Сначала покажи diff-план/список затрагиваемых заголовков перед применением». 
7) Приложите ЧЕК-ЛИСТ приемки (ниже) — я сверю по пунктам перед финалом.

---

### Политика терминов (пример по вашему проекту)
- Не переводить: `Production`, `Staging`, `Development`, `Testing`, `CI/CD`, `Git Workflow`, `Release`, `Branching Strategy`, `API`, `Routes`, `Authentication`, `Sanctum`.
- Переводить описательный текст вокруг (объяснения, инструкции, примечания) на русский.
- Имена эндпоинтов, HTTP-методы, коды и JSON — не переводим и не переписываем.
- Имена файлов/путей/команд — не переводим.

Если нужен обратный вариант (всё на русском), просто замените правила.

---

### Что именно писать в задаче (шаблон)
Скопируйте, отредактируйте и пришлите как ТЗ.

```
Задача: Привести README.md к единой языковой политике.

Область:
- Файл: README.md
- Разделы: ВСЕ заголовки H2–H4, подписи <summary>, названия таблиц/блоков.

Языковая политика:
- Заголовки и подписи разделов — EN (единообразно по всему документу)
- Основной текст (описания, инструкции, замечания) — RU

Глоссарий:
- НЕ переводить: Production, Staging, Development, Testing, API, Routes, Authentication, Sanctum, CI/CD, Git Workflow, Release, Branching Strategy, License
- Переводить: описания шагов, комментарии «как запустить», «как тестировать», «примечания»

Ограничения:
- Не менять код-блоки, команды, HTTP-примеры, JSON, пути, роуты, версии, шильдики
- Не удалять и не переименовывать разделы
- Сохранить Markdown-структуру и теги <details>/<summary>

Стиль:
- Нейтральный технический, без жаргона. Краткость, точность, без разговорных форм.

Порядок выполнения:
1) Покажи «план изменений»: список всех заголовков, которые будут изменены (до/после)
2) Дождись подтверждения
3) Применяй изменения

Чек-лист приёмки:
- [ ] Все заголовки на EN; тело разделов на RU
- [ ] Ни один код-блок/JSON/HTTP-ответ не изменён
- [ ] <details>/<summary> работают, разметка не поломана
- [ ] Ссылки и якоря не сломаны
- [ ] Нет смешения языков в заголовках
```

---

### Примеры правильных формулировок (чтобы не было двусмысленности)
- Плохо: «Переведи README нормально».  
  Хорошо: «В README.md заголовки EN, содержимое RU; не трогать кодовые блоки; пришли список изменяемых заголовков до применения».

- Плохо: «Сделай mc».  
  Хорошо: «В README.md: заменить все H2–H4 заголовки на английские эквиваленты; тело оставить на русском; термин Production не переводить; показать diff-план».

- Плохо: «Переведи сложные части».  
  Хорошо: «Перевести только описательные параграфы в разделах Environments, Running the Application, Tests; заголовки оставить на английском; кодовые блоки не трогать».

---

### Как я буду действовать по этому плану
1) Пройду README.md, соберу список затрагиваемых заголовков и подпунктов.  
2) Выведу «план изменений» списком: До → После (без изменения файлов).  
3) После вашего «ОК» применю изменения единым патчем.  
4) Быстрая самопроверка по чек-листу.  
5) Выложу результат и краткий отчёт: что изменено и что принципиально не трогалось.

---

### Частые ошибки и как их избежать
- Смешение языков в заголовках — решается явной политикой «заголовки EN / тело RU» (или наоборот, но единообразно). 
- Случайный трог кода/HTTP-примеров — запрещаем в ТЗ менять содержимое блоков тройными бэктиками. 
- «Перевод по одному слову» вне контекста — запрещаем точечный перевод терминов без правки всего класса элементов (например, всех заголовков одного уровня).
- Непонятые аббревиатуры («mc») — всегда расшифровывать в задаче, иначе стопор.

### Быстрый мини-шаблон (если вы очень занят)
Если нет времени расписывать подробно — пришлите одной строкой, и этого хватит:

```
README.md: заголовки EN, текст RU; Production/CI/CD не переводить; код-блоки/JSON/HTTP-примеры не менять; покажи список заголовков до правки.
```

### Готовная форма для будущих запросов
Скопируйте, вставьте, замените нужные слова:

```
Сделай перевод в README.md по политике: заголовки — EN, основной текст — RU.
Не переводить термины: Production, Staging, API, Routes, Authentication, Sanctum, CI/CD.
Кодовые блоки, JSON и HTTP-примеры — не менять.
Сначала пришли план: список заголовков до/после. После подтверждения — применяй.
```

Если хотите наоборот (всё на русском или всё на английском) — просто поменяйте одну строку «политики», всё остальное остаётся прежним.
